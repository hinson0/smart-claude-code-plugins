실행 단계 (반드시 순서대로 수행하며 건너뛸 수 없습니다):

## 단계 2: 커밋

⚠️ 아래의 commit skill에는 중요한 의미 분석 단계(3단계)가 포함되어 있습니다. 완전히 실행하십시오 — 파일-목적 분석 테이블과 분할 결정을 출력한 후에 git 명령을 실행하십시오.

@../commit/SKILL_KO.md

⚠️ 검증: 위의 커밋 결과는 의미 분석의 그룹 분류와 일치해야 합니다. 모든 파일을 한 번에 커밋했지만 분석에서 여러 type/목적이 표시된 경우, 즉시 중단하고 다시 수행하십시오.

- 작업 영역에 변경 사항이 없으면 이 단계를 건너뛰고 단계 3으로 직접 진행합니다.

**모든 커밋 완료 후, `git status --short`를 실행하여 남은 변경 사항이 있는지 확인합니다.** 아직 커밋되지 않은 파일이 있으면 해당 파일에 대해 커밋 단계를 자동으로 다시 실행합니다 — 사용자에게 확인하지 않습니다. 작업 트리가 깨끗해질 때까지 반복합니다. 사용자가 push를 실행하는 의도는 모든 변경 사항을 푸시하는 것입니다.

## 단계 3: 푸시

### 3.1 origin 설정 여부 확인

실행: `git remote get-url origin 2>/dev/null`

- 설정된 경우: `git push -u origin HEAD`를 직접 실행하고 3.3으로 건너뜁니다.
- 설정되지 않은 경우: 3.2를 계속합니다.

### 3.2 GitHub 원격 저장소 자동 생성 및 연결

순서대로 실행:

1. `gh` CLI 로그인 확인: `gh auth status`
   - 로그인되지 않은 경우 "먼저 `gh auth login`을 실행하세요"라는 메시지를 출력하고 **중단**합니다.

2. 저장소 이름 읽기: `basename $(git rev-parse --show-toplevel)`

3. 현재 GitHub 사용자 이름 읽기: `gh api user --jq .login`

4. 원격에 동일한 이름의 저장소가 이미 존재하는지 확인: `gh repo view <사용자명>/<저장소명> 2>/dev/null`
   - 이미 존재하는 경우: 직접 연결하고 6단계로 건너뜁니다.
   - 존재하지 않는 경우: 5단계를 계속합니다.

5. GitHub 저장소 생성 (기본값 비공개):
   ```
   gh repo create <저장소명> --private --source=. --remote=origin
   ```

6. 이미 존재하지만 연결되지 않은 경우 수동으로 remote 추가:
   ```
   git remote add origin https://github.com/<사용자명>/<저장소명>.git
   ```

### 3.3 푸시 실행

```
git push -u origin HEAD
```

## 출력 결과

성공 시 표시:
1. 단계 1 검사 결과 요약.
2. 단계 2에서 실제 사용된 모든 commit message (변경 사항이 있는 경우).
3. 푸시 대상 브랜치 및 결과.
4. 최종 `git status` (작업 영역이 깨끗한지 확인).

실패 시 표시:
- 실패가 발생한 단계와 스텝.
- 구체적인 오류 메시지.
- 다음으로 실행할 수 있는 수정 명령.

## 제약 사항

- git config를 수정하지 않습니다.
- `--amend`, `--force`, `--no-verify`를 사용하지 않습니다.
- 이번 커밋과 직접 관련된 명령만 실행하며, 추가적인 리팩토링이나 파일 수정을 하지 않습니다.

学习模式让用户亲手编写代码中有意义的部分。它的目的不是拖慢进度——而是有些决策（核心逻辑、数据建模）恰恰是用户学到最多、领域知识最重要的地方，所以把这些部分交还给用户，而不是替他写完。每类代码交还多少，**按项目可配置**。

本 skill 是一个开关加一小块配置。它在 `.smart/settings.json` 里保存机器可读的状态，**并且**把真正的行为规则镜像写入 `.claude/CLAUDE.local.md`——后者会被 Claude Code 在每次会话开始时自动载入上下文。settings 文件是真值源；`CLAUDE.local.md` 里的注释块是让规则无需每次重新调用 skill 也能持续生效的机制。

## 参数

| `$0`                       | 行为                                                       |
| -------------------------- | ---------------------------------------------------------- |
| `1`                        | **开启** —— 把 `learning` 置 1，并注入参与规则块。         |
| `0`                        | **关闭** —— 把 `learning` 置 0，并移除注入块。             |
| `config` `[bucket=NN ...]` | **配置** —— 更新 `learning_ratios` 中的一个或多个占比。    |
| _（空）_                   | **查看状态** —— 报告开/关、生效占比，以及任何不一致。      |

若 `$0` 不是上述值，打印用法并停止。

## settings 文件：`.smart/settings.json`

这就是 distill 用的**同一个**项目级文件——全局只有一个。务必读-改-写：保留所有不属于你的键（尤其 distill 的 `knowledges_dir`）。本 skill 拥有两个键：

- **`learning`** —— `0 | 1`，开关。
- **`learning_ratios`** —— 可选对象；用户亲手编写的各类占比，整数百分比。缺失的键回退到下表默认值。

  | 桶（bucket）  | 含义                                       | 默认 |
  | ------------- | ------------------------------------------ | ---- |
  | `boilerplate` | 用户亲写的样板 / 重复性代码                | `30` |
  | `core`        | 用户亲写的有意义逻辑（为*下限*）           | `60` |
  | `database`    | 建表 / Schema / 迁移 / 索引                | `100`|

整个 `.smart/` 目录是约定的个人配置位置，通常已被 git-ignore，所以学习状态留在本地、绝不提交。

## CLAUDE.local.md 注释块

本 skill 只管理标记之间的区域；`.claude/CLAUDE.local.md` 里其余内容都属于用户，绝不可触碰：

## 步骤

### 1）解析项目根目录并读取参数

- 项目根：`git rev-parse --show-toplevel`；若不在 git 仓库中，回退到当前工作目录。
- 读取 `$0`，按下面的 开启 / 关闭 / 配置 / 查看状态 分支处理。
- 任何时候需要**生效占比**时，从 settings 文件读 `learning_ratios`，缺失的桶用默认值补齐。

### 2）开启（`$0 == 1`）

1. `mkdir -p <root>/.smart`。
2. **写入状态。** 若 `<root>/.smart/settings.json` 已存在，读取它，把 `learning` 置为 `1`，写回时**保留其他所有键**。若不存在，新建为 `{"learning": 1}`。
3. **确保 `.claude/CLAUDE.local.md` 存在且已被 git-ignore。** 若文件不存在，按 `@../local/SKILL.md` 引导它（连同它的 `.gitignore` 条目）。若已存在，保持其内容不动。
4. **注入参与规则块**（见下文"参与规则块"），并把生效占比代入。需幂等：
   - 若 `<!-- SMART:LEARNING:BEGIN -->` … `<!-- SMART:LEARNING:END -->` 标记已存在，把标记之间（含标记）整体替换——这会在占比变更或版本升级后刷新旧块。
   - 否则把该块追加到文件末尾，前面隔一个空行。
5. **确保 `.smart/` 被 git-ignore。** 执行 `git check-ignore -q .smart/settings.json`。若失败（尚无规则忽略它），用 Edit/Write 工具把单独一行 `.smart/` 追加到 `<root>/.gitignore`——`.gitignore` 不存在则新建，绝不重复，绝不改写无关内容。已忽略 `.smart/` 的仓库里这是空操作。
6. **立即生效。** 规则现已激活。声明在*本*会话剩余时间里你也会遵守它们，而不仅是未来会话。
7. **报告**：`learning = 1`、生效占比、块是新增还是更新、git-ignore 是新加还是早已生效。

### 3）关闭（`$0 == 0`）

1. **写入状态。** 读取 `<root>/.smart/settings.json`（必要时新建），把 `learning` 置为 `0`，保留其他键。保留 `learning_ratios`，让配置值留到下次开启。
2. **移除注入块。** 若 `<root>/.claude/CLAUDE.local.md` 存在且含标记，删除从 `<!-- SMART:LEARNING:BEGIN -->` 到 `<!-- SMART:LEARNING:END -->`（含标记）的整段，外加相邻一个空行，避免留下空隙。其余每一行保持不动。**绝不删除 `CLAUDE.local.md` 文件本身**，也不动 `.gitignore`。
   - 若文件或标记不存在，则没什么可删——如实说明。
3. **停止应用。** 说明参与规则在本会话剩余时间不再适用。
4. **报告**：`learning = 0`、块是已移除还是本就不存在。

### 4）配置（`$0 == config`）

1. **解析其余 token** 为 `bucket=NN` 形式（`boilerplate`、`core`、`database`）；`NN` 为 `0..100` 的整数。遇未知桶或越界值，给一行简短用法并停止。
2. **若没有任何有效键值对**，打印当前生效占比与用法（`/smart:learning config boilerplate=40 core=70 database=100`），并说明也可直接编辑 `.smart/settings.json`。然后停止——不要臆测数值。
3. **合并并持久化。** 读取 settings（必要时新建），把给定的桶并入 `learning_ratios`（局部更新——未触及的桶保留原值），写回时保留其他所有键。
4. **若处于开启则重注入。** 若 `learning == 1`，刷新 `CLAUDE.local.md` 块（步骤 2.4），让新占比立即生效。若处于关闭，仅持久化——值在下次开启时应用。
5. **报告** 占比 旧 → 新，以及实时块是否已刷新。

### 5）查看状态（无参数）

1. 读取 `<root>/.smart/settings.json`；文件缺失或缺 `learning` 键时按 `0` 处理。
2. 解析生效占比（配置值，缺口用默认值补齐）。
3. 检查 `.claude/CLAUDE.local.md` 是否含受管理的标记。
4. 报告 `learning = 0 | 1`、生效占比，以及块是否存在。**提示不一致**：若状态为 `1` 但块缺失（或状态为 `0` 但块仍在），如实指出，并提示可通过重跑 `/smart:learning 1` 或 `/smart:learning 0` 对齐。

## 参与规则块

在标记之间注入下面这段内容，并把 `{boilerplate}`、`{core}`、`{database}` 替换为生效占比。**按用户的工作语言把正文本地化**——即其 `CLAUDE.local.md` 与对话所用的语言。下面的中文是要落地的规范版本；只改语言，不改含义。

```markdown
<!-- SMART:LEARNING:BEGIN -->
## 学习模式（已开启）

用户希望亲自编写代码中有意义的部分。实现时，把下列部分交还给用户来写，而不是你自己写完。百分比是用户亲手编写的占比（配置在 `.smart/settings.json` → `learning_ratios`）：

- **样板 / 脚手架**：用户写 ~{boilerplate}%。其余你来写，但把用户那一份留作标注清晰的 TODO 桩。
- **核心业务逻辑**：用户至少写 ~{core}% 的关键判断点（算法、分支、错误处理策略、数据结构）。把这些留为 TODO 桩，先讲清权衡，然后停下来请用户补全。
- **数据库（建表 / Schema / 迁移 / 索引）**：用户设计 ~{database}%。为 100% 时你一行都不写——只提供需求与约束。

遇到这些部分时：先搭好周边上下文，写清函数签名与注释，在对应位置标 TODO，然后**停下来**，等用户提交代码后再继续。每次请求都要框定为一个真实的设计决策，而非杂活——说明在哪个文件、哪个位置，以及需要权衡什么。
<!-- SMART:LEARNING:END -->
```

## 约束

- `.smart/settings.json` 与 distill 共用：**始终读-改-写，绝不覆盖**。保留 `knowledges_dir` 及任何不属于你的键。
- 只触碰 `CLAUDE.local.md` 的标记区域；绝不破坏用户的其他笔记。
- 所有改动幂等——重复执行同一命令得到相同最终状态，绝不产生重复块或重复的 `.gitignore` 行。
- `.gitignore` 用 Edit/Write 工具处理（追加单独一行 `.smart/`）；绝不整文件改写。`.smart/` 已忽略时跳过。
- 占比为 `0..100` 的整数百分比；`core` 为*下限*，`database` 默认 100（用户完全设计 Schema）。
- 输出语言与用户对话一致。
